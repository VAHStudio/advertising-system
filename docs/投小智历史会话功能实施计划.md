## 1. 项目概述

### 1.1 背景
投小智当前存在 Dify 和 Custom 两种对话模式，但缺乏完善的历史会话管理功能。用户每次打开都需要重新开始对话，无法查看和继续之前的会话。

### 1.2 目标
- 支持双模式（DIFY/CUSTOM）独立的历史会话管理
- 每个模式展示各自的最近10条历史对话，支持向上滚动加载更多
- 保存 CUSTOM 模式的消息历史到 MySQL 数据库
- 使用 Redis 缓存优化性能

### 1.3 技术栈
- **后端**: Spring Boot 4.0.3, Java 21, MyBatis, MySQL 8.0, Redis
- **前端**: Vue 3, TypeScript, Vite, Tailwind CSS
- **AI**: Dify API, Custom Agent Service

## 2. 架构设计

### 2.1 数据模型

#### ai_conversation 表（扩展）
```sql
- id (PK, BIGINT, AUTO_INCREMENT)
- user_id (VARCHAR(100), INDEX) - 用户ID
- conversation_id (VARCHAR(100), UNIQUE) - 会话唯一标识
- mode (VARCHAR(20)) - AI模式: DIFY | CUSTOM
- title (VARCHAR(200)) - 会话标题
- status (INT, DEFAULT 1) - 状态: 1-活跃, 0-归档
- message_count (INT, DEFAULT 0) - 消息数量统计
- last_message_preview (VARCHAR(200)) - 最后消息预览
- created_at (TIMESTAMP) - 创建时间
- updated_at (TIMESTAMP) - 更新时间
- last_message_at (TIMESTAMP, INDEX) - 最后消息时间

索引:
- idx_user_mode_time (user_id, mode, last_message_at DESC)
- idx_conversation_lookup (conversation_id, mode)
```

#### ai_conversation_message 表（新增）
```sql
- id (PK, BIGINT, AUTO_INCREMENT)
- conversation_id (VARCHAR(100), INDEX) - 关联会话ID
- role (VARCHAR(20)) - 角色: user | assistant
- content (TEXT) - 消息内容
- thinking (TEXT) - 思考过程（Custom模式）
- tool_calls (JSON) - 工具调用信息
- metadata (JSON) - 额外元数据
- created_at (TIMESTAMP) - 创建时间

索引:
- idx_conversation_time (conversation_id, created_at DESC)
- idx_conversation_role (conversation_id, role)
```

### 2.2 Redis 缓存策略

```
Key Patterns:
- ai:conv:{userId}:{mode} → Sorted Set (score=lastMessageAt)
  存储会话列表，按最后消息时间排序
  TTL: 1小时

- ai:msg:{conversationId} → List (LPUSH新消息)
  存储消息列表
  TTL: 30分钟

- ai:active:{userId} → String
  当前活跃会话ID
  TTL: 24小时
```

### 2.3 前后端交互流程

```
1. 打开投小智
   ↓
2. 前端根据当前模式请求 /api/ai-assistant/conversations?mode={mode}&page=0
   ↓
3. 后端查询Redis缓存，未命中则查询MySQL
   ↓
4. 返回会话列表（最近10条）
   ↓
5. 用户点击某个会话
   ↓
6. 前端请求 /api/ai-assistant/conversations/{id}/messages?page=0
   ↓
7. 后端返回消息历史（分页）
   ↓
8. 用户向上滚动加载更多历史消息
   ↓
9. 前端请求下一页消息
```

## 4. 接口文档

### 4.1 获取会话列表
```http
GET /api/ai-assistant/conversations?mode={mode}&page={page}
Authorization: Bearer {token}

Parameters:
- mode (required): DIFY | CUSTOM
- page (optional): 页码，从0开始，默认0

Response:
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 1,
      "conversationId": "conv-xxx",
      "mode": "CUSTOM",
      "title": "客户咨询",
      "messageCount": 12,
      "lastMessagePreview": "好的，我来帮您查询...",
      "lastMessageAt": "2026-03-13T10:30:00"
    }
  ]
}
```

### 4.2 创建会话
```http
POST /api/ai-assistant/conversations?mode={mode}
Authorization: Bearer {token}

Parameters:
- mode (required): DIFY | CUSTOM

Response:
{
  "code": 200,
  "message": "success",
  "data": {
    "conversationId": "custom-uuid",
    "mode": "CUSTOM"
  }
}
```

### 4.3 获取消息历史
```http
GET /api/ai-assistant/conversations/{conversationId}/messages?page={page}
Authorization: Bearer {token}

Parameters:
- page (optional): 页码，从0开始，默认0

Response:
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 1,
      "role": "user",
      "content": "帮我查询杭州的空位",
      "createdAt": "2026-03-13T10:25:00"
    },
    {
      "id": 2,
      "role": "assistant",
      "content": "正在为您查询...",
      "thinking": "用户要查询...",
      "createdAt": "2026-03-13T10:26:00"
    }
  ]
}
```

### 4.4 更新标题
```http
PUT /api/ai-assistant/conversations/{conversationId}/title?title={title}
Authorization: Bearer {token}

Response:
{
  "code": 200,
  "message": "success"
}
```

### 4.5 删除会话
```http
DELETE /api/ai-assistant/conversations/{conversationId}
Authorization: Bearer {token}

Response:
{
  "code": 200,
  "message": "success"
}
```

## 5. 验收标准

### 功能验收
- [ ] 打开投小智自动加载当前模式的最近10条历史会话
- [ ] 会话列表可按最后消息时间排序
- [ ] 向上滚动会话列表自动加载更多（分页）
- [ ] 点击会话显示消息历史，支持向上滚动加载更多
- [ ] 可新建 DIFY/CUSTOM 模式会话
- [ ] 可删除历史会话（带确认）
- [ ] 可重命名会话标题
- [ ] 可搜索历史会话
- [ ] 切换模式时自动切换对应的历史列表

### 性能验收
- [ ] 会话列表加载 < 500ms（缓存命中）
- [ ] 会话列表加载 < 1s（缓存未命中）
- [ ] 消息历史加载 < 300ms
- [ ] Redis 缓存命中率 > 80%
- [ ] 支持 1000+ 会话不卡顿

### 安全验收
- [ ] 用户只能访问自己的会话
- [ ] 敏感数据（思考过程）正确存储
- [ ] SQL 注入防护
- [ ] XSS 防护

## 6. 风险与应对

| 风险 | 级别 | 应对措施 |
|------|------|----------|
| 数据库迁移失败 | 高 | 先在测试环境验证，备份数据 |
| Redis 连接不稳定 | 中 | 降级到直接查数据库，优雅降级 |
| CUSTOM 消息量大 | 中 | 设置单会话消息上限（如500条） |
| 性能不达标 | 中 | 增加二级缓存，优化查询 |
